## Complexity Assessment Protocol

Before every dispatch, assess estimated_complexity:

- low: single file, < 50 lines, well-defined → no checkpoint needed
- medium: 2-3 files, or 50-200 lines, or moderate ambiguity → checkpoint required
- high: 4+ files, or > 200 lines, or cross-module, or > 5 min estimated → checkpoint + execution plan + consider decomposition

estimated_complexity controls COMMUNICATION PROTOCOL STRICTNESS ONLY; model selection is handled by the escalation chain.

Always include estimated_complexity in the dispatch payload.

## Skip-to-Heavy Gate

The governor MAY bypass the standard executor and dispatch directly to `executor-heavy.toml` when ALL of the following are true:
- estimated_complexity = high
- task involves >= 5 files across >= 3 distinct modules
- prior evidence (weakness registry or recent escalation history) shows this task class consistently fails with the standard executor

When triggered:
- set attempt_number = 2
- dispatch directly to `executor-heavy.toml`
- log `SKIP_TO_HEAVY dispatch_ref={ref} reason={reason}`

When in doubt, do NOT skip; start with the standard executor.

Before using skip-to-heavy or heavy escalation, confirm the model named in `.codex/agents/executor-heavy.toml` is actually supported by the active runtime.
If it is unavailable, record a runtime-alignment blocker and do NOT silently substitute another executor while claiming heavy escalation occurred.

Default model is ALWAYS the standard executor (`executor.toml`) unless skip-to-heavy gate triggers.

## Task Decomposition Protocol

Decompose a task into batches when ANY of these apply:
- >= 3 independent units of work
- Expected wall-clock time > 5 minutes
- >= 4 files with cross-module dependencies
- >= 3 sequential steps without intermediate validation

Batch rules:
- Each batch <= 2 independent items or <= 1 complex item
- Each batch gets its own dispatch_ref: {parent_ref}/B01, /B02, ...
- Each batch must produce a checkpoint artifact
- Review each checkpoint before dispatching the next batch
- If any batch exhausts escalation (3 attempts), stop the parent task

## Diagnosis-first Dispatch Protocol

Use `task_track = diagnosis` for:
- artifact comparison

- bounded instrumentation
- failure-surface isolation

Use `task_track = patch` for:
- localized implementation against an already-identified failure surface

Stay in `diagnosis` track when the exact failing key path, source schema, or expected artifact is still unclear.
Move to `patch` track only when a diagnosis artifact or retry handoff makes the target explicit.

Runtime patch authorization inside the active lane:

## Escalation Decision Protocol

When an executor's output fails Tier 3 eval:

1. Check attempt_number in dispatch payload
2. If attempt_number < 3:
   - Build escalation_context with prior eval feedback
   - Re-dispatch using executor-heavy (gpt-5.4, xhigh reasoning) if attempt >= 2
   - Increment attempt_number, set escalated = true
   - Log: ESCALATION attempt_number={n}
3. If attempt_number = 3 and still failing:
   - STOP autonomous execution
   - Compile full escalation report (all 3 attempts, eval results, files touched)
   - Present to human
   - Log: ESCALATION_EXHAUSTED

## Handling Executor Escalation Requests

When an executor produces an escalation request artifact:
- Read the trigger and context
- Decide: clarify and re-dispatch, absorb as direct-work (20-line limit), or escalate to human
- Do NOT count escalation requests as failed attempts
- Log: EXECUTOR_ESCALATION_REQUEST

## Retry Handoff Protocol

When retrying a failed diagnosis or patch task, include `retry_handoff` with:
- exact failing validator or gate
- exact failing artifact path
- exact failing key path
- exact source schema or contract reference
- exact expected artifact reference
- expected value summary
- observed value summary

If you cannot provide the exact failing key path, source schema, and expected artifact, do NOT send another vague `patch` retry. Revert to `diagnosis` track first.
